EasyTicket: Steps to Run the Project So Far

1. Open the project folder

   cd /Users/ayushchhetry/ticket

2. Create and activate a Python virtual environment

   python3 -m venv .venv
   source .venv/bin/activate

3. Install dependencies

   pip install -r requirements.txt

4. Create your local environment file

   cp .env.example .env

5. Use mock AI mode for local development

   Keep these values in .env:

   MOCK_AI_MODE=true
   USE_MOCK_AI=true

   This lets the project run without a Gemini API key and avoids token/API costs.

6. Optional: configure live Gemini mode

   Only do this when you want real Gemini API calls:

   GEMINI_API_KEY=your_api_key_here
   MOCK_AI_MODE=false
   USE_MOCK_AI=false

   Do not commit .env. It is ignored by Git.

7. Initialize and seed the local database

   .venv/bin/python scripts/seed_data.py

   The default SQLite database is:

   data/tickets.db

8. Run the FastAPI backend

   .venv/bin/uvicorn app.main:app --reload

   Backend URL:

   http://127.0.0.1:8000

   Health check:

   http://127.0.0.1:8000/health

9. Run the Streamlit frontend in a second terminal

   cd /Users/ayushchhetry/ticket
   source .venv/bin/activate
   EASYTICKET_BACKEND_URL=http://127.0.0.1:8000 .venv/bin/streamlit run frontend/app.py

   Frontend URL:

   http://127.0.0.1:8501

10. Combined backend + frontend manual test

   Terminal 1:

   .venv/bin/uvicorn app.main:app --reload

   Terminal 2:

   EASYTICKET_BACKEND_URL=http://127.0.0.1:8000 .venv/bin/streamlit run frontend/app.py

   In Streamlit:

   Select a customer.
   Enter a ticket message.
   Click Create Ticket.
   Copy or keep the created ticket ID.
   Click Analyze Ticket.
   Confirm category, subcategory, confidence, priority, assigned team, reason, and analysis source are displayed.
   Open Ticket Details and confirm ticket history appears.

11. API endpoint examples

   Health:

   curl http://127.0.0.1:8000/health

   Create ticket:

   curl -X POST http://127.0.0.1:8000/tickets \
     -H "Content-Type: application/json" \
     -d '{"customer_id":"CUST-1001","message":"My payment was deducted but my subscription is inactive."}'

   List tickets:

   curl "http://127.0.0.1:8000/tickets?status=open&page=1&page_size=20"

   View ticket:

   curl http://127.0.0.1:8000/tickets/TICKET_ID_HERE

   Analyze ticket:

   curl -X POST http://127.0.0.1:8000/tickets/TICKET_ID_HERE/analyze

   Update ticket:

   curl -X PATCH http://127.0.0.1:8000/tickets/TICKET_ID_HERE \
     -H "Content-Type: application/json" \
     -d '{"status":"in_progress"}'

   Manual escalation:

   curl -X POST http://127.0.0.1:8000/tickets/TICKET_ID_HERE/escalate \
     -H "Content-Type: application/json" \
     -d '{"reason":"Customer is blocked and needs engineering review.","team":"engineering"}'

   Ticket history:

   curl http://127.0.0.1:8000/tickets/TICKET_ID_HERE/history

16. Manual RAG system check

   The base RAG setup is local and does not require an API key. It reads text files from:

   /Users/ayushchhetry/ticket/knowledege base docs

   Run this from the project root:

   .venv/bin/python -c "from app.models.customer import Customer; from app.models.ticket import Ticket; from app.services.rag_search_service import search_knowledge_base; from app.services.missing_information_service import detect_missing_information; from app.services.sentiment_service import analyze_sentiment; from app.services.escalation_decision_service import decide_escalation; from app.services.escalation_packet_service import generate_escalation_packet; import json; message='API returns 401 unauthorized in production for workspace WS-123 in Chrome.'; ticket=Ticket(customer_id='CUST-1001', message=message); customer=Customer(customer_id='CUST-1001', name='Demo Customer', plan='enterprise', previous_tickets=2); rag=search_knowledge_base(message); missing=detect_missing_information(message, 'technical'); sentiment=analyze_sentiment(message); decision=decide_escalation(ticket_message=message, category='technical', priority='high', assigned_team='engineering', retrieval_confidence=rag.confidence, customer_context={'plan': customer.plan, 'previous_tickets': customer.previous_tickets}, missing_information=missing); packet=generate_escalation_packet(ticket=ticket, customer=customer, classification='technical', priority='high', rag_result=rag, escalation_decision=decision); print(json.dumps({'rag_sources': rag.sources, 'rag_confidence': rag.confidence, 'missing_information': missing, 'sentiment': sentiment.__dict__, 'decision': decision.model_dump(), 'packet': packet.model_dump()}, indent=2))"

   Expected manual signals:

   rag_sources should include api_authentication_errors.txt.
   rag_confidence should be greater than 0.
   sentiment should show urgency because production is mentioned.
   decision should request missing information or escalate depending on missing fields and priority.
   packet should include internal_note and customer_reply.

17. Manual text normalization check

   The original ticket message is stored as submitted. Normalized text is used only for matching, retrieval, sentiment, and missing-information checks.

   Run this from the project root:

   .venv/bin/python -c 'from app.services.text_normalization_service import normalize_user_text; from app.services.rag_search_service import search_knowledge_base; from app.services.rule_based_router import classify_ticket; from app.services.sentiment_service import analyze_sentiment; message="pls help i can'\''t login asap"; print(normalize_user_text(message)); print(classify_ticket(message)); print(analyze_sentiment("pls fix asap production blocked")); print(search_knowledge_base("i paid for pro but my sub isn'\''t active").sources)'

   Expected manual signals:

   Normalized text should include please, cannot, and as soon as possible.
   Routing should classify the login message as account-related.
   Sentiment should show urgency for asap/production/blocked.
   RAG sources should include subscription_activation.txt for the messy subscription message.
